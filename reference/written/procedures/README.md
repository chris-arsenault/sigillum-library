# procedures/ — the runbooks

How the work is actually executed. Not an ordered series — each is its own runbook.

- **`dsl/`** — **START HERE FOR NEW PRODUCTION DSL COMPOSITION.** One coherent Ruby DSL procedure for
  fresh composition, including research routing, material design, section writing, flow/interleaving,
  whole-piece revision, export, and closeout. This is the default for new scores whose source of truth
  is `production_piece`.
- **`unified_procedure.md`** — **START HERE FOR LEGACY SYMPHONY/PYTHON WORK.** The canonical process:
  Stage 0–7, the rule taxonomy (Hard / Doctrine / Judgment / Execution), the gate suite. This remains
  the canonical symphony/Python procedure and the source of many musical laws the DSL fork imports.
- **`orchestration_procedure.md`** — the multi-agent fork of `unified_procedure`: fan a movement
  across fresh section-agents that compose → ornament → inspect → refine. Python/symphony-shaped in
  its mechanics.
- **`merge_procedure.md`** — fusing several complete drafts of the same movement into one: analyze each
  draft, carry every draft's high-water mark forward, author a fresh frame + motion-curve contract, fan
  out the sections, then measure the realization to confirm the richness landed.
- **`post_merge_cypher_fractalization_procedure.md`** — the whole-movement final pass after merge
  validation: local cypher/fractalization incidents that break section-block seams without broad-brush
  repainting.
- **`section_repair_procedure.md`** — re-running / repairing a single section a movement run produced.
- **`card_writing_procedure.md`** — how to research and author a texture-card (research-led, verbatim).
- **`composition_method.md`** — the per-section rationale annex (the A.1–A.9 steps: cast, line,
  counterline, bass-as-line, inner motion, orchestrate, vary returns, composed transitions, gates).
